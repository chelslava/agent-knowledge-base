# AGENTS.md — Правила работы агентов с базой знаний

> Этот файл — **главный контракт** для всех ИИ-агентов (Claude Code, Cursor Agent,
> OpenCode, Gemini CLI, Aider, Roo Code и др.), работающих с этой базой знаний.
> Читается **первым** при старте любой сессии.

---

## 0. Назначение системы

База знаний — это **долговременная память** между проектами и сессиями. Её цели:

- хранение знаний в формате Markdown;
- накопление опыта между проектами;
- переиспользование готовых решений;
- автоматическая документация проектов;
- интеграция с [md-semantic-search](https://github.com/chelslava/md-semantic-search);
- предотвращение дублирования информации.

---

## 1. Базовые правила (ОБЯЗАТЕЛЬНЫ К ИСПОЛНЕНИЮ)

1. **База знаний — основной источник контекста.** Перед выполнением задачи
   агент обязан прочитать релевантные документы из базы знаний, а не полагаться
   только на содержимое репозитория кода.
2. **Семантический поиск перед созданием заметки.** Прежде чем создать новую
   заметку, выполни семантический поиск через `md-semantic-search` (см. раздел 4).
   Если знание уже существует — **дополни** существующий документ, а не создавай новый.
3. **Запрет дубликатов.** Не создавай заметки с дублирующимся содержанием.
   Одна тема — один канонический документ (см. [[knowledge-rules]]).
4. **Obsidian-ссылки.** Все связи между заметками оформляются через
   `[[имя-файла-без-расширения]]`. Ссылайся щедро — связи важнее изоляции.
5. **Автообновление `index.md`.** После создания/изменения/удаления документа
   обнови [[index]] (карту базы знаний).
6. **Автообновление `CHANGELOG.md`.** Каждое значимое изменение фиксируется в
   [[CHANGELOG]] (дата, тип, краткое описание, ссылки).
7. **Автообновление `taxonomy.md`.** При появлении новых тегов/категорий обнови
   [[taxonomy]].
8. **Актуальность документации проектов.** Документы проекта поддерживаются в
   синхроне с реальным состоянием кода и решений.

---

## 2. Жизненный цикл задачи (workflow)

Каждая задача выполняется по следующему алгоритму:

1. **Читать документацию проекта.** Перед любой задачей открой документы проекта:
   `project-overview.md`, `project-index.md`, `architecture.md`, `decisions.md`,
   `requirements.md`, `technical-debt.md`.
2. **Семантический поиск** через [md-semantic-search](https://github.com/chelslava/md-semantic-search)
   по всей базе знаний (`shared/`, `projects/`, `inbox/`).
3. **Использовать `shared/`** как источник общих, переиспользуемых знаний
   (паттерны, рецепты, чек-листы, гайды).
4. **Не копировать** одинаковую информацию между проектами — ссылайся на
   канонический документ через `[[...]]`.
5. **Использовать канонические документы.** У каждой темы есть один источник
   истины (canonical). Остальные документы ссылаются на него.
6. **Создавать связи** между заметками через Obsidian-ссылки `[[...]]`.

> Подробные правила — в [[knowledge-rules]].

## 3. Куда что складывать

| Что | Куда | Пример |
|-----|------|--------|
| Сырые входные данные, недооформленные мысли | `inbox/` | вырезки, черновики |
| Переиспользуемые между проектами знания | `shared/` | паттерны, рецепты, гайды |
| Документация конкретного проекта | `projects/<project>/` | см. структуру проекта ниже |
| Шаблоны документов | `templates/` | заготовки |
| Конфигурация семантического поиска и индексы | `semantic/` | конфиг md-semantic-search |

Новый документ всегда создаётся из соответствующего шаблона в [[index|templates/]].

## 4. Семантический поиск (md-semantic-search)

Перед созданием новой заметки и при поиске контекста используй
[md-semantic-search](https://github.com/chelslava/md-semantic-search).

CLI инструмента — **`mdss`** (Node ≥18, локально, без API-ключей; кросс-язычный
поиск — русский запрос находит и англоязычные документы). Модель этой базы —
**`bge-m3`** (закреплена в обёртках `kb-index.*`). Проще всего — через обёртки
(сами подставляют корень, папку индекса, модель и исключения; фолбэк на `npx`):

```bash
# Индексация (после изменений / по расписанию)
pwsh knowledge-base/semantic/kb-index.ps1     # или: bash .../kb-index.sh

# Поиск перед созданием заметки
pwsh knowledge-base/semantic/kb-search.ps1 "формулировка темы заметки"
```

Прямой вызов и флаги — в [[README|semantic/README]].

Правило: **если верхние результаты поиска уже покрывают тему — дополняй
найденный документ, не создавай дубликат.** Жёсткого порога score нет; у `bge-m3`
релевантное сидит ~0.70–0.77, но с **широким отрывом** от нерелевантного —
смотри на разрыв топ-1 и верхний сниппет, а не на абсолютный cosine.

## 5. Структура проекта (обязательная)

Каждый проект в `projects/<project-name>/` содержит:

- `project-overview.md` — обзор, цели, контекст
- `project-index.md` — карта документов проекта
- `architecture.md` — архитектура (canonical)
- `architecture-diagram.md` — диаграммы (Mermaid)
- `requirements.md` — требования
- `roadmap.md` — дорожная карта
- `decisions.md` — журнал решений (ADR)
- `api-reference.md` — справочник API
- `development-log.md` — лог разработки
- `release-notes.md` — заметки о релизах
- `technical-debt.md` — технический долг
- `wiki/` — внутренняя вики проекта (свободные заметки)
- `raw/` — сырые материалы по проекту

Регистрация проекта — в [[projects-registry]].

## 6. Правила оформления заметок

- Формат — Markdown с YAML-frontmatter (см. шаблоны).
- Заголовок H1 один на документ.
- Теги — из контролируемого словаря [[taxonomy]].
- Связи — `[[wikilink]]`; для именованной ссылки `[[имя|подпись]]`.
- Даты — в ISO-формате `YYYY-MM-DD`.
- Каждый документ указывает `canonical: true|false` и при необходимости ссылку
  на канонический источник.
