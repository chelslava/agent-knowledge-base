# Интеграция с ИИ-агентом (Claude Code и др.)

Как сделать так, чтобы агент **сам** читал базу знаний перед задачей, пополнял её
по контракту и **версионировал** изменения в git без ручных коммитов.

Принцип универсален: любой агент, который умеет читать `AGENTS.md` (Cursor,
Gemini CLI, Aider, OpenCode, Roo Code…), подхватит контракт. Ниже — конкретная
настройка для **Claude Code**.

> Везде замените `<KB_ROOT>` на абсолютный путь к вашей копии базы, например
> `C:/Users/you/agent-knowledge-base` (Windows) или
> `/home/you/agent-knowledge-base` (POSIX).

---

## 1. Глобальный блок в `CLAUDE.md`

Чтобы агент в **любом** проекте знал про базу и работал по её правилам, добавьте
этот блок в свой пользовательский `CLAUDE.md` (обычно `~/.claude/CLAUDE.md`).
Это обезличенный каркас — отредактируйте путь под себя.

````markdown
# 🧠 База знаний — основной инструмент памяти (ОБЯЗАТЕЛЬНО)

> Единая долговременная память между проектами и сессиями — Markdown-база знаний
> по адресу `<KB_ROOT>`. Она — **основной источник и хранилище информации**, а не
> репозиторий кода и не файловая память агента.

## Контракт базы
1. В начале **любой** сессии/задачи прочитай контракт базы:
   `<KB_ROOT>/AGENTS.md` (главный контракт) и
   `<KB_ROOT>/knowledge-rules.md` (детальные правила). Работай **строго** по ним.
2. База — основной источник контекста. Перед задачей читай релевантную
   документацию из базы (`shared/`, `projects/<name>/`), а не только код.
3. Если информации по проекту в базе **нет** — заведи её по всем правилам:
   зарегистрируй проект в `projects/projects-registry.md`, создай структуру
   `projects/<name>/` из шаблонов `templates/`, заполни и свяжи документы.

## Перед созданием/изменением заметки
4. **Антидубликат-протокол.** Перед созданием новой заметки выполни семантический
   поиск. Из корня `<KB_ROOT>`:
   - поиск: `pwsh semantic/kb-search.ps1 "формулировка темы"`
     (или `bash semantic/kb-search.sh "…"`);
   - переиндексация после правок: `pwsh semantic/kb-index.ps1`.
   Если тема уже покрыта — **дополняй** найденный документ или добавь
   перекрёстную ссылку, новый дубль не создавай.
5. **Single Source of Truth.** Одна тема — один канонический документ
   (`canonical: true`); остальное ссылается через `[[...]]`, текст не копируется.
   Переиспользуемое между проектами знание живёт в `shared/`.
6. Оформление: Markdown + YAML-frontmatter из `templates/`, один H1, теги из
   `taxonomy.md`, даты `YYYY-MM-DD`, щедрые `[[wikilink]]`, секция «См. также».

## Обязательная синхронизация мета-файлов
7. Обнови `index.md` (карта), `CHANGELOG.md` (дата/тип/описание/ссылки),
   `taxonomy.md` (если появились новые теги). Поле `updated:` — на текущую дату.
8. После изменений в коде/решениях актуализируй документы проекта
   (`architecture.md`, `decisions.md` как ADR, `development-log.md`,
   `technical-debt.md`).

> Полные чек-листы — в `AGENTS.md` §7 и `knowledge-rules.md`. При расхождении
> этих правил с файлами базы — **файлы базы являются источником истины**.
````

## 2. Авто-коммит базы знаний (Stop-hook)

Скрипт `tools/kb-autocommit.ps1` (и POSIX-вариант `.sh`) коммитит изменения базы
после **каждого** ответа агента. Корень репозитория он вычисляет от собственного
расположения (`tools/` → корень), поэтому путь-независим — работает из любой
сессии на машине.

Подключите его как **Stop-hook** в `~/.claude/settings.json`.

**Windows (PowerShell):**

```json
{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "pwsh -NoProfile -ExecutionPolicy Bypass -File \"<KB_ROOT>/tools/kb-autocommit.ps1\""
          }
        ]
      }
    ]
  }
}
```

**POSIX (bash):**

```json
{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "bash \"<KB_ROOT>/tools/kb-autocommit.sh\""
          }
        ]
      }
    ]
  }
}
```

Поведение скрипта:

- нет изменений → тихий выход (`git status` ~0.1 с, накладные минимальны);
- есть изменения → `git add -A` + коммит `kb: auto <дата время>`;
- если настроен `origin` — фоновый `git push` (fire-and-forget, сессию не
  блокирует, ошибки сети игнорируются — следующий коммит дошлёт накопленное);
- любые ошибки подавляются, скрипт **никогда не роняет** сессию.

> Хотите приватную историю? Привяжите `origin` к **приватному** репозиторию —
> публичным остаётся только этот шаблон-каркас, а ваши реальные знания пушатся
> в приватный.

## 3. Проверка интеграции

1. Откройте новую сессию агента в любом проекте.
2. Попросите что-нибудь записать в базу (например: «заведи в shared заметку о
   паттерне X»).
3. Агент должен: выполнить семантический поиск → создать заметку из шаблона →
   обновить `index.md`/`CHANGELOG.md`.
4. После ответа Stop-hook создаст коммит `kb: auto …` в репозитории базы:

```bash
git -C "<KB_ROOT>" log --oneline -3
```

## Другие агенты

- **Cursor / Aider / Gemini CLI / OpenCode:** достаточно, чтобы агент прочитал
  `AGENTS.md` в начале работы — контракт самодостаточен. Авто-коммит подключите
  аналогичным хуком/скриптом вашего инструмента или коммитьте базу вручную.
- **Без агента вовсе:** база — обычный Obsidian-vault. Открывайте, пишите,
  ищите через `kb-search`, коммитьте сами.
