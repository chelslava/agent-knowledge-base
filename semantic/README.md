# 🔎 Semantic — Семантический поиск (md-semantic-search / `mdss`)

Локальный семантический поиск по базе знаний на
[md-semantic-search](https://github.com/chelslava/md-semantic-search) (CLI `mdss`,
Node ≥18, на `@xenova/transformers`). **Без API-ключей и облака** — всё считается
на машине. **Обязателен перед созданием новой заметки** (правило 2 в [[AGENTS]],
антидубликат-протокол в [[knowledge-rules]]).

> Инструмент изначально проектировался под двуязычную (RU/EN) инженерную вики и
> поддерживает **кросс-язычный** поиск: запрос на русском находит документ на
> английском и наоборот.
>
> **Модель этой базы — `bge-m3`** (`Xenova/bge-m3`, 1024d, ~2.3 ГБ) — выбрана за
> максимальное кросс-язычное разделение (по RESEARCH.md — самый широкий отрыв
> правильного результата). Обёртки `kb-index.*` пиннят её по умолчанию.
> (Дефолт самого инструмента — более лёгкий `e5-base`, 768d, ~280 МБ.)

---

## Быстрый старт (обёртки этой базы)

В папке `semantic/` лежат готовые скрипты — они сами подставляют корень базы,
папку индекса (`semantic/.mdss`) и исключения (`semantic/**`, `templates/**`).
Работают из любой директории; если нет глобального `mdss`, используют `npx`.

```powershell
# Windows / PowerShell
pwsh knowledge-base/semantic/kb-index.ps1                 # построить/обновить индекс
pwsh knowledge-base/semantic/kb-search.ps1 "тема заметки" # поиск
```

```bash
# POSIX / bash
bash knowledge-base/semantic/kb-index.sh
bash knowledge-base/semantic/kb-search.sh "тема заметки"
```

**Антидубликат-правило:** если верхние результаты уже покрывают тему —
**дополняй** найденный документ, а не создавай новый (см. [[knowledge-rules]]).

## Прямой вызов `mdss`

```bash
# установка (любой вариант)
npx md-semantic-search --help          # без установки
npm install -g md-semantic-search      # даёт команды mdss и md-semantic-search

# из корня knowledge-base/:
mdss index  --db . --index-dir semantic/.mdss --ignore "semantic/**" --ignore "templates/**"
mdss search --db . --index-dir semantic/.mdss "формулировка темы"
mdss search --db . --index-dir semantic/.mdss --json --k 8 "тема"   # для агентов
mdss search --db . --index-dir semantic/.mdss --semantic "тема"     # только вектор
mdss models                                                          # список моделей
```

> Модель читается из индекса при поиске — повторять `--model` в `search` не нужно.

## Как это работает (по исходникам инструмента)

- **Обход файлов:** рекурсивно `*.md` / `*.markdown`. **Dot-папки и dot-файлы
  пропускаются автоматически** (`.git`, `.mdss`, `.cache` — индексировать их не
  нужно и нельзя случайно).
- **`--ignore <glob>`** (повторяемый): заякоренный glob по относительному пути от
  `--db`. `*` = один сегмент пути, `**` = через `/`. Примеры: `"semantic/**"`,
  `"templates/**"`, `"**/archive/**"`, `"log.md"`.
- **Чанкинг:** по Markdown-заголовкам; секции > ~1400 символов дробятся по пустым
  строкам. Поиск возвращает релевантную **секцию** с её заголовком.
- **Заголовок документа:** берётся из frontmatter `title:` или первого `# H1` →
  поэтому шаблоны базы с `title:` индексируются осмысленно.
- **Ранжирование:** гибрид — косинус (смысл) + лексическое совпадение (точные
  идентификаторы), слитые через Reciprocal Rank Fusion. `--semantic` отключает
  лексику. Стоп-слова RU/EN отфильтрованы.
- **Инкрементальность:** per-file md5 в `.mdss/.hashes.json`; неизменённые файлы
  переиспользуют эмбеддинги. Смена модели → полная переиндексация.
- **Индекс:** `semantic/.mdss/vectors.json` (+ `.hashes.json`). Один JSON-файл,
  векторного БД не нужно (корпус малый).

## Выдача

Человекочитаемо: `N. [cos 0.xxx] <Заголовок>` + путь `file › heading` + сниппет.
`--json` → массив объектов `{ file, title, heading, cosine, score, snippet }`,
где `cosine` — близость по смыслу (0…1), `score` — итоговый RRF-балл.

> **Ориентир по `cosine` зависит от модели** (по замерам RESEARCH.md):
> - `bge-m3` (модель этой базы): релевантное сидит **ниже по абсолюту (~0.70–0.77)**,
>   но с **широким отрывом** от нерелевантного (margin ~0.065) — разделение чёткое.
> - `e5-base`: релевантное **~0.80+**, но отрывы у́же (~0.02–0.03).
>
> Поэтому **жёсткого порога нет** — это мягкий сигнал. Смотри не на абсолютное
> число, а на **разрыв между топ-1 и следующими** и на верхний сниппет; держи
> гибрид (без `--semantic`), чтобы ловить точные идентификаторы.

## Выбор модели

| Нужно | Модель | Команда |
|-------|--------|---------|
| **Максимум кросс-язычного качества (~2.3 ГБ)** | **`bge-m3`** *(выбор этой базы)* | — *(дефолт обёрток)* |
| Лёгкий дефолт инструмента, быстро, ~280 МБ | `e5-base` | `kb-index.* --model e5-base` |
| Минимальный вес, запросы одноязычные/буквальные | `e5-small` | `--model e5-small` |

`e5-small` **не рекомендуется** для кросс-язычного поиска (по RESEARCH.md ранжирует
нерелевантное выше нужного). Можно передать и любой `Xenova/*` id в `--model`.

> Сменить модель базы: задай `MDSS_MODEL` или передай `--model` в `kb-index.*`.
> **Смена модели → полная переиндексация** (размерности и префиксы несовместимы) —
> инструмент пересоберёт индекс целиком автоматически.

## Когда мигрировать на sqlite-vec

Сейчас индекс — один файл `vectors.json` (брутфорс-косинус). Это осознанный выбор:
формат хранилища **не влияет на качество** (его определяют модель и эмбеддинги),
только на скорость загрузки/обновления и масштаб. Для текущего размера JSON
оптимален. Переходить на [`sqlite-vec`](https://github.com/asg017/sqlite-vec)
(векторное хранилище + KNN в одном файле) стоит, когда выполнится **хотя бы одно**:

1. **Объём:** счёт чанков уходит за **~10–20k** (десятки проектов × сотни секций) —
   полная загрузка JSON в RAM и его перезапись при каждом `index` начнут давить.
2. **Фильтрация по метаданным:** нужен поиск «только в `projects/<x>/`», по
   `type/decision`, по тегам [[taxonomy]] — это естественно ложится на SQL `WHERE`,
   а на JSON делается костылём.
3. **Конкурентность:** несколько агентов индексируют/читают параллельно.

До этих порогов SQLite даёт **нулевой выигрыш**. При переходе предпочесть
`sqlite-vec`/ANN (`hnswlib`), а не «голый» SQLite, и реализовать это опцией
бэкенда в самом инструменте (репозиторий `chelslava/md-semantic-search`).

## Авто-переиндексация (Windows Task Scheduler)

Индекс нужно пересобирать после правок (`mdss` инкрементальный — без изменений
прогон ≈ **0.1с**, модель даже не грузится, поэтому частые запуски почти бесплатны).
Автоматизировано задачей планировщика.

```powershell
# Установить задачу (однократно; права админа не нужны — под текущим пользователем)
pwsh knowledge-base/semantic/install-task.ps1            # каждые 30 минут
pwsh knowledge-base/semantic/install-task.ps1 -Minutes 15

# Управление
Get-ScheduledTask   -TaskName KB-Semantic-Reindex
Start-ScheduledTask -TaskName KB-Semantic-Reindex        # прогнать сейчас
Unregister-ScheduledTask -TaskName KB-Semantic-Reindex -Confirm:$false   # удалить
```

- Раннер `kb-reindex-task.ps1` вызывает `kb-index.ps1`, пишет `.logs/reindex.log`
  (с ротацией) и сам добавляет npm-prefix в `PATH`, чтобы найти глобальный `mdss`.
- Задача работает «при входе пользователя». Для запуска без входа —
  пересоздай с `-User`/`-Password` или `-RunLevel`.
- Предусловия (выполнены при установке): `npm install -g md-semantic-search`
  (стабильная команда `mdss`) и единый кэш моделей в `semantic/.cache`
  (чтобы bge-m3 не перекачивался ~каждый раз).

## Артефакты в папке

- [[mdss.env.example]] — пример env (`MDSS_DB` / `MDSS_INDEX_DIR` / `MDSS_CACHE_DIR` / `MDSS_MODEL`).
- `kb-index.{ps1,sh}` · `kb-search.{ps1,sh}` — обёртки индексации/поиска (ASCII-safe).
- `kb-reindex-task.ps1` — раннер для планировщика; `install-task.ps1` — установщик задачи.
- `.mdss/` — индекс; `.cache/` — кэш моделей; `.logs/` — логи. Всё генерируется,
  в VCS не коммитится (см. `knowledge-base/.gitignore`).
